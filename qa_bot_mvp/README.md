# Game QA 자동화 — 봇 플레이 기반 이상탐지

봇이 게임을 자동으로 플레이해 궤적을 만들고, 정상 플레이만 학습한 모델이
비정상 플레이를 잡아낸다. 게임을 직접 만들지 않고, 이미 있는 게임 환경
(MiniGrid, ViZDoom)을 어댑터로 붙인다.

## 아키텍처: 어댑터 + 게임 무관 코어

```
[게임별 어댑터] --표준 Episode/Step--> [게임 무관 코어]
 MiniGrid (격자)                        스키마 / 특징 / 규칙 / 이상탐지 / 시계열 / 리포트
 ViZDoom (연속)                         (어떤 게임이든 재사용)
```

새 게임을 붙인다는 것은 표준 포맷을 뱉는 어댑터 하나를 추가한다는 뜻이다.
코어는 수정하지 않는다. ViZDoom 어댑터를 붙일 때 코어 코드 변경은 0줄이었다.

코어는 어댑터가 넣는 게임별 특징의 이름과 의미를 모른다. 그냥 숫자로 받아
벡터에 싣는다. 그래서 어댑터가 특징을 0개 넣든 100개 넣든 코어는 그대로다.

## 3층 탐지 구조

| 층 | 하는 일 | 방법 | 잡는 것 |
|---|---|---|---|
| 1. 규칙 (하드 오라클) | 불변식 위반을 확정 | `Episode.meta`의 물리 파라미터 기반 검사 | teleport, stuck, out_of_bounds, invalid_coord |
| 2. 집계 이상탐지 | 판 단위 이상 점수 | IsolationForest, 정상만 학습 | under_explore, low_entropy 같은 미묘한 이상 |
| 3. 시계열 | 이상의 위치(스텝)를 지목 | 윈도우 예측 모델 (sklearn MLP, window=5) | oscillate_mid 같은 순서 이상 |

규칙이 이상탐지를 이긴다. 규칙 위반은 확정이므로 리포트 최상단에 오고,
그 아래가 이상 점수 내림차순이다. 규칙은 정상 플레이에서 발동하면 안 된다
(오탐 0 원칙). 두 게임 모두 정상 오탐 0을 유지한다.

## 실행

```bash
pip install gymnasium minigrid scikit-learn numpy joblib vizdoom
python run.py            # MiniGrid (격자 세계)
python run_vizdoom.py    # 실제 Doom (연속 좌표, 헤드리스)
```

torch는 필요 없다. LSTM을 시도할 때 설치했지만 최종 모델은 sklearn 윈도우
모델이다. 이유는 `NOTE_lstm_vs_window.md`에 있다.

## 파이프라인

1. 정상 봇이 게임을 플레이해 궤적(Episode)을 수집한다.
2. 정상 데이터만으로 학습한다 (집계 이상탐지 + 시계열 스텝 모델). 결함은 학습에 쓰지 않는다.
3. 검사 대상 판을 3층(규칙 / 집계 / 시계열)으로 검사한다.
4. 결과를 한 리포트로 합친다. 규칙 위반 먼저, 그다음 이상 점수 내림차순.

결함 주입(`demo/`)은 학습이 아니라 평가에만 쓴다. 모델은 결함을 본 적이 없다.
그래서 미지의 버그도 원리상 잡을 수 있다.

## 결과

`python run.py` / `python run_vizdoom.py` 재현값이다.

| 항목 | MiniGrid (격자) | Doom (연속) |
|---|---|---|
| 규칙층 teleport | 60/60 | 주입 불가 (엔진 제약) |
| 규칙층 stuck | 60/60 | 40/40 |
| 규칙층 정상 오탐 | 0/100 | 0/40 |
| 집계층 low_entropy | 60/60 (1.00) | 39/39 (1.00) |
| 집계층 under_explore | 9/35 (0.26) | 3/11 (0.27) |
| 집계층 정상 오탐 | 0.12 | 0.175 |
| 시계열 oscillate_mid 지역화 | 93/103 (0.90) | 0/4 (0.00) |
| 종합 탐지율 (규칙 OR 집계) | 0.913 | 0.900 |
| 종합 정상 오탐 | 0.120 | 0.175 |
| 코어 코드 수정 | 기준 | 0줄 |

모델 학습은 1회 13.5초가 걸리고, 저장한 모델을 로드해 실행하면 0.1초다.

### 수치를 읽는 법

집계층 수치의 분모는 60이나 40이 아니라 "규칙이 놓친 판"이다. under_explore는
이동을 억제하는 결함이라 정지 구간이 길어져 `stuck` 규칙에 먼저 걸리는 경우가
많다 (MiniGrid 25/60, Doom 29/40). 이미 규칙이 확정한 판은 집계층이 다시 볼
이유가 없으므로, 남은 판만 분모로 잡았다.

## 핵심 발견

- **이상탐지는 정상 분포가 촘촘한 특징일수록 잘 작동한다.** low_entropy는 두 게임
  모두 100퍼센트 잡힌다. 정상 봇의 행동 분포가 촘촘해서 벗어남이 뚜렷하다.
  반대로 under_explore는 26퍼센트다. 정상 무작위 플레이 자체가 공간 커버리지
  편차가 커서, 좁게 돈 판이 정상의 하위 꼬리와 겹친다.
- **격자는 선명하고 연속은 노이즈다.** 이 패턴이 프로젝트 전반에 반복된다.
  Doom 시계열은 어떤 모델을 써도 약하다 (0에서 20퍼센트). 연속 환경의 좌표
  노이즈가 예측 오차 신호를 삼킨다.
- **LSTM이 윈도우 모델보다 나빴다** (MiniGrid 90퍼센트에서 15퍼센트로 하락).
  예측 오차 기반 탐지는 모델이 "정상은 모델링하되 이상은 못 하는" 스위트스팟에서
  최적인데, LSTM은 너무 강해서 주기적 이상까지 예측해 버린다. 자세한 내용은
  `NOTE_lstm_vs_window.md`에 있다.
- **특징은 게임 무관이 아니다.** 스키마와 탐지 모델은 그대로 재사용되지만, 특징
  계산은 격자와 연속 좌표의 스케일 차이 때문에 게임별 파라미터가 필요하다.
  어댑터가 `move_eps`, `cell_size`를 `meta`로 선언하고 코어는 한 벌을 유지한다.
- **코어의 특징 개수 자동 확장을 검증했다** (8개에서 10개, 108개까지). 코어 코드
  수정은 0줄이었다.

## 시스템 포지셔닝과 한계

정직하게 밝힐 부분이다.

- 이 시스템은 정상 데이터를 **받는** 것이 아니라 봇이 **생성한다**. modl.ai 같은
  상용 서비스는 회사가 제공한 사람 플레이 데이터를 쓴다.
- 따라서 여기서 말하는 정상은 "무작위 봇의 정상"이지 "사람의 정상"이 아니다.
  실제 배포에서는 회사 제공 데이터를 파일로 등록하는 경로를 쓴다.
- 규칙층이 다루는 것은 어떤 이동형 게임에나 공통인 최소 불변식뿐이다. 게임별
  스펙(퀘스트, 아이템 등)은 개발자가 코드에 심는 assertion의 몫이다.
- 결함 주입은 물리적으로 분리되어 있다. `adapters/`는 배포용이고 `demo/`는
  시연용이며, 어댑터는 결함 모듈을 import하지 않는다.

## 리포트 철학

리포트는 정답 라벨을 담지 않는다. 운영에서는 무슨 버그였는지 알 수 없기 때문이다.
모델이 실제로 낼 수 있는 것(anomaly_score, flagged, 규칙 위반, 이상 스텝 위치)과
관측 가능한 사실(seed, outcome, features)만 담는다.

버그 종류도 담지 않는다. 이상탐지 모델에는 종류라는 개념 자체가 없다.
리포트가 하는 일은 트리아지다. 제일 수상한 판부터 정렬해 주고, 사람이 features를
근거로 판단한다. 리포트는 단정하지 않고 재료를 정렬해 줄 뿐이다.
