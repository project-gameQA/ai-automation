# OpenArena QA 자동화

봇이 사람 대신 게임을 플레이하는 동안, 시스템이 봇의 상태를 관측해 버그를 자동으로 탐지한다.
대상은 OpenArena(퀘이크3/ioquake3 계열 오픈소스 아레나 FPS)다.

핵심은 **액터와 오라클의 분리**다. 봇은 플레이만 하고, 버그 판정은 별도의 오라클이 맡는다.
이렇게 나누면 액터를 무엇으로 바꾸든(내장 봇, 스크립트, 강화학습) 판정 로직은 그대로 쓸 수 있다.

---

## 무엇을 잡는가

오라클을 세 층으로 나눈다. 각 층이 잡는 것의 성격이 다르다.

| 층 | 잡는 것 | 방식 | 상태 |
|---|---|---|---|
| ① 하드 인바리언트 | **물리적으로 불가능한 상태** | 규칙 5종 | 완료 |
| ② 이상탐지 | **가능하지만 정상이 아닌 패턴** | 비지도 학습 | 완료 |
| ③ 성능/크래시 워치독 | **게임이 느려지거나 멈춤** | 시뮬/실제 시간 비교 + 프로세스 관측 | 완료 |

이 구분은 실패에서 나왔다. 초기에 "체력이 최대치를 넘으면 버그"라는 규칙을 넣었는데
OpenArena에서는 오버힐이 정상이라 오탐이 쏟아졌다. 여기서 얻은 원칙이
**"하드 인바리언트는 불가능을 잡고, 드물다는 이상탐지의 몫"**이다.

### ① 하드 인바리언트 (규칙 기반)

| 규칙 | 판정 |
|---|---|
| `fell_through_floor` | 맵 바닥 아래로 내려감 |
| `out_of_bounds` | 맵 경계를 벗어나거나 천장을 넘음 |
| `health_out_of_range` | 어떤 메커니즘으로도 도달 불가능한 체력 |
| `impossible_velocity` | 물리적 상한을 넘는 속력 |
| `stuck` | **이동 입력이 있는데** 위치가 변하지 않음 |

`stuck`이 이 프로젝트의 성격을 잘 보여 준다. "오래 정지"로 잡으면 조준·매복 중인 정상 봇이
걸린다. 대신 **의도(이동 입력)와 결과(위치 변화)의 불일치**를 보면 오탐이 사라진다.
이를 위해 게임 소스에서 `move_input`을 텔레메트리에 실어 보내도록 계측을 추가했다.

### ② 이상탐지 (비지도 학습)

정상 플레이만 학습하고 거기서 벗어난 정도를 점수로 낸다. 무엇이 이상인지 미리 알려 주지 않는다.

- **입력**: 60초 창의 활동 특징 3개 (발사 비율, 체력 손실률, 체력 유지율)
- **모델**: IsolationForest
- **임계값**: 학습 데이터 점수의 5백분위 (오탐률 기준선을 직접 통제하기 위함)
- **출력**: 이상 점수 + **어떤 특징이 몇 표준편차 벗어났는지**

결론이 아니라 단서다. 점수만 주면 "검사하라"는 말은 되지만 "무엇을 검사하라"는 말은
되지 않으므로, 기여 특징을 함께 낸다.

### ③ 성능/크래시 워치독

앞의 둘은 봇을 보지만 이쪽은 **게임 프로세스 자체**를 본다. 봇이 정상으로 보여도 게임이
버벅이거나 멈춰 있으면 그것도 버그다.

텔레메트리의 `time` 은 시뮬레이션 시간이라 성능을 담지 못한다. 서버가 실제로 멈춰도
`level.time` 은 고정 간격으로 흐른다. 그런데 **서버는 실제 시각도 알고 있으므로**, 둘을
비교하면 게임 수정 없이 성능을 잴 수 있다.

    실효 서버 틱 = (Δ시뮬레이션 시간 / 프레임 주기) / Δ실제 시간

여기에 프로세스 관측(psutil)을 더하면 **행과 종료가 구분된다.** 텔레메트리만으로는 둘 다
"기록이 멈췄다"로 같아 보인다.

| 텔레메트리 | 프로세스 | 판정 |
|---|---|---|
| 멈춤 | 살아 있음 | **진행 없음.** 행이거나 메뉴·점수판 구간이다 |
| 멈춤 | 없음 | 종료. 크래시인지 정상인지는 사람이 판단 |
| 멈춤 | 살아 있음 + 파일 잘림 직후 | 맵 로딩(정상, 경보 억제) |

게임을 켜지 않았거나 지난 기록을 읽는 중인 상태는 이상이 아니므로 판정하지 않는다.
밀린 기록은 실제 시간보다 시뮬레이션 시간이 훨씬 빨리 흐르는 것으로 구분한다.

주의: 여기서 재는 것은 **서버 틱**이지 클라이언트 렌더 FPS 가 아니다. 화면 프레임은
텔레메트리에 없고 재려면 cgame 계측이 필요하다.

## 검증 결과

### 하드 인바리언트

실제 게임에서 결함 주입기 4종(`qa_inject_health` / `oob` / `fall` / `stuck`)으로 규칙 5종을
양성·음성 모두 검증했다. 바닥 관통은 좌표 조작이 아니라 **콜리전 제거**(tracemask에서
`CONTENTS_SOLID`·`CONTENTS_BOTCLIP` 제거)로 실제 낙하를 만들어, 화면 시연과 검출을 동시에
성립시켰다.

### 집계

하드 인바리언트는 매 프레임 판정하므로 5초짜리 낙하가 100건으로 불어난다. 사람이 검토할
단위는 사건 하나다. `(봇, 규칙)` 단위로 0.5초 이내 연속 탐지를 하나로 묶는다.

```
원시 탐지 237건 → 6개 사건
  BOT1 fell_through_floor   3.00~ 4.95s (1.95s,  40프레임)  최저 z=-2107.0
  BOT2 stuck                8.00~12.95s (4.95s, 100프레임)  최대 정지 시간=9.95
```

### 이상탐지

정상 세션(봇 8마리, 610초, 448창)으로 학습하고 네 세션을 채점했다.

| 세션 | 이상 판정 | 판정의 의미 |
|---|---|---|
| 정상 (대조군, 학습 미사용) | **8.2%** | 오탐률. 기준선 5%의 1.6배 |
| 저중력 (`g_gravity 500`) | 0.0% | 정답. 활동 면에서는 실제로 정상 |
| 반동 증폭 (`g_knockback 1500`) | 0.0% | 정답. 활동 면에서는 실제로 정상 |
| **봇 실력 최저 (`g_spSkill 1`)** | **100.0%** | 전부 탐지 |

봇 실력 세션에서 하드 인바리언트는 **0건**이다. 규칙을 하나도 어기지 않는데 시스템이
이상을 짚어낸다는 것이 ② 오라클의 존재 이유다. 주 원인은 발사 비율(−4.0 ~ −5.0σ)로 지목됐다.

## 실행

**처음 세팅한다면 `docs/setup.md` 를 먼저 본다.** 이 저장소만으로는 동작하지 않는다.
게임 설치, 게임 C 소스에 계측 적용, QVM 빌드·배포가 선행돼야 한다.

### 한 번에

```
run_qa.bat        가상환경 활성화 + API 서버 + 프론트엔드
stop_qa.bat       종료
```

`run_qa.bat` 위쪽의 `CONDA_ENV`와 `TELEMETRY`(게임이 기록하는 파일 경로)를 한 번 맞춰 둔다.

### 게임

```
openarena.exe +set fs_game qa +set sv_pure 0
```

콘솔(`~`)에서 결함 주입: `\qa_inject_health 1` … `0`으로 끔.
매치가 중간에 끝나지 않게: `\fraglimit 0` `\timelimit 0`

### 오프라인 분석

```
python run_invariants.py <텔레메트리>              사건 단위 출력
python run_invariants.py <텔레메트리> --raw        집계 전 원시 탐지

python tools/train_anomaly.py <정상 세션...>       이상탐지 모델 학습
python tools/score_anomaly.py <세션>               채점 (오탐률·탐지율)

python tools/calibrate_bounds.py <정상 플레이>     맵 경계 보정
python tools/measure_interval.py <텔레메트리>      프레임 간격 측정
python tools/measure_perf.py --seconds 180         워치독 기준선 (서버 실행 중에)
python tools/extract_features.py <A> --compare <B> 특징 분포 비교
```

### 이 저장소에 없는 것

게임 쪽은 별도 준비가 필요하다. 무엇이 없고 왜 없는지, 어떻게 갖추는지는 `docs/setup.md` 1절에 있다.

| 없는 것 | 대신 있는 것 |
|---|---|
| OpenArena 게임 본체 | `docs/setup.md` 3절의 설치 절차 |
| 게임 C 소스(`gamecode`) | 변경 16건 전체가 `docs/gamecode_instrumentation.md` 에 위치·이유와 함께 기록 |
| 빌드·배포 배치 파일 | `docs/setup.md` 6절의 템플릿 |
| 텔레메트리·세션 데이터, `node_modules`, 학습된 모델 | 실행·빌드하면 생긴다 |

게임 소스 변경은 **세 파일에 흩어진 작은 편집 16건**이다(`g_main.c`, `g_local.h`, `g_active.c`).

## 설계에서 배운 것

실패한 시도와 그로부터 나온 판단을 기록으로 남겼다. 상세는 `docs/process.md` 참조.

**"드물다"와 "잘못됐다"는 다르다.** 체력 규칙 오탐 사건에서 얻었다. 같은 함정이 이상탐지
특징 설계에도 있어, 좌표를 직접 학습하지 않는다. 봇이 잘 안 가는 맵 구석은 드물지만 정상이다.

**이상이 창 하나가 아니라 분포에 있을 수 있다.** 실력 낮은 봇은 거의 안 싸우는데, 정상
플레이에도 5초 동안 한 발도 안 쏘는 창은 흔하다. 창 길이별 탐지율은 5초 4.4%, 30초 64.6%,
60초 95.2%였다. 모델이나 학습의 문제가 아니라 **표현의 문제**였다.

**특징을 더 넣는 것이 항상 도움이 되지는 않는다.** 60초 창에 이동 특징 11개를 함께 쓰면
오탐이 8.7% → 16.8%로 두 배가 되고 탐지율은 100% → 90.3%로 떨어졌다. 정상 세션끼리도
18.8%를 이상으로 판정해, 신호가 아니라 세션 간 흔들림을 잡고 있었다.

**물리 파라미터 교란은 시험지로 부적합하다.** 중력·마찰·반동을 세 번 시도했으나 모두
실패했다. 약하면 봇이 적응해 행동이 그대로고, 강하면 경계를 벗어나 하드 규칙이 먼저 잡는다.
그 사이 지대가 거의 없다. 반면 봇 실력 하향은 정확히 그 지대에 있었다.

**시험지에 맞춰 특징을 고치지 않는다.** 저중력을 100% 잡는 특징(하강 가속도)을 찾았으나
채택하지 않았다. 정상 두 판에서 값이 소수점까지 같아 특징이 아니라 **상수**였고,
"중력이 800이 아니면 이상"은 규칙 한 줄이지 ML이 필요한 일이 아니다.

**재현 가능한 파생물은 버려도 되고, 결론은 버리면 안 된다.** 원시 탐지는 텔레메트리만 있으면
다시 만들 수 있으므로 최근 것만 보관한다. 사건은 시스템이 내린 결론이므로 전부 보관하고
파일로도 남긴다. 이상 점수도 파생물이라 파일로 남기지 않는다.

**표시의 정당성은 표시가 아니라 그 뒤의 동작에서 나온다.** 정적 스냅샷 시절에는 LIVE 표시를
제거했다가, 서버가 실제로 텔레메트리를 따라가게 된 뒤에 되살렸다.

## 알려진 한계

- **맵 의존**: 경계·바닥·과속 규칙은 맵마다 보정이 필요하다. 체력·끼임은 무관하다.
  현재 한 맵 기준이며, 맵 이름을 텔레메트리에 추가하고 맵별 설정 표를 두는 것이 확장 방향이다.
- **짧은 이동 이상을 이상탐지가 잡지 못한다**: 끼임·진동·맴돌기는 ② 오라클의 범위 밖이다.
  짧은 창 척도가 동작한다는 근거를 확보하지 못해 넣지 않았다.
- **오탐률 8.2%**: 기준선 5%의 1.6배다. 학습 데이터가 세션 하나뿐이라 세션 간 흔들림을
  정상 범위로 흡수하지 못했다. 정상 세션을 여러 개 모아 학습하면 줄어들 여지가 있다.
- **워치독 임계값이 미검증이다**: 성능 저하 판정 기준(목표 틱의 75%)과 멈춤 판정 시간(5초)은
  정한 상수이며 이 환경에서 정상 플레이의 흔들림을 재 보지 않았다. `tools/measure_perf.py` 로
  기준선을 잡은 뒤 조정해야 한다.
- **"진행 없음"이 행과 메뉴를 구분하지 못한다**: 매치 종료 후 메인 메뉴, 점수판 구간, 진짜 행이
  모두 "텔레메트리가 멈췄고 프로세스는 살아 있다"로 같아 보인다. 구분하려면 게임의 현재 상태를
  텔레메트리에 실어 보내야 한다.
- **클라이언트 렌더 FPS 는 재지 못한다**: 워치독이 재는 것은 서버 틱이다. 화면 프레임은
  텔레메트리에 없고 cgame 계측이 필요하다.
- **원인·결과가 얽힌 사건이 따로 잡힌다**: 자유낙하 중 속도가 커져 과속이 함께 뜬다.
- **진행 중 사건은 프로세스가 갑자기 죽으면 유실된다**: 닫힌 사건만 즉시 파일에 기록된다.
- **세션 폴더는 자동 정리되지 않는다**: 봇 4마리 기준 실제 플레이 시간당 약 46MB가 쌓인다.

## 게임 쪽 계측

게임 로직 모듈(`qagame`)에만 손댔다. 엔진은 건드리지 않았고, 빌드는 QVM 모드로 해서 스톡
0.8.8 엔진에 얹는다. 변경한 파일은 셋(`g_main.c`, `g_local.h`, `g_active.c`), 편집은 16건이다.

| 넣은 것 | 무엇을 위해 |
|---|---|
| 텔레메트리 훅 (`ClientEndFrame`) | 봇 상태를 매 서버 프레임 JSONL로 기록. `SVF_BOT` 검사로 봇만 |
| 결함 주입 cvar 4종 | 규칙을 재현 가능하게 검증 |
| 입력 필드 (`move_input`, `attack`) | 끼임 규칙의 "의도 vs 결과" 판정, 이상탐지의 발사 비율 |

**주입 순서에 규칙이 있다.** 상태를 오염시키는 주입은 텔레메트리 기록보다 **위**에 둔다
(오염값이 기록되도록). 바닥 관통만 예외로 `ClientThink_real`에 둔다(Pmove 전에 tracemask 를
바꿔야 하므로). 그리고 바닥 관통은 좌표를 조작하지 않고 **콜리전을 제거**해 실제로 떨어뜨린다.
화면 시연과 검출이 동시에 성립한다.

**계측이 담지 못하는 것도 정리해 뒀다**(`docs/gamecode_instrumentation.md` 3-1). 실제 시각,
명중·데미지, 맵 이름, 게임 상태(매치/점수판), 봇 이름이다. 각각 무엇을 못 하게 되는지와
그럼에도 넣지 않은 이유를 적었다. 대표적으로 텔레메트리의 `time` 은 시뮬레이션 시간이라
성능을 담지 못하는데, 서버가 실제 시각과 비교하는 방식으로 게임 수정 없이 우회했다.

## 하드웨어 제약

Intel HD Graphics 4600(2013 Haswell 내장 그래픽), RAM 16GB 환경에서 개발했다. GPU가 병목이라
UE5 등은 쓸 수 없어 OpenArena 0.8.8 + QVM 모드를 택했다. QVM은 게임에 동봉된 도구(lcc/q3asm)로
빌드하므로 최신 컴파일러와 옛 코드의 충돌도 피할 수 있다.

이 제약이 설계에 영향을 준 부분이 있다. 실시간 감시에서 파일 전체를 다시 읽지 않고 꼬리만
읽는 것, 원시 탐지를 전부 보관하지 않는 것, 대시보드가 그리는 행 수를 제한하는 것이 그렇다.
