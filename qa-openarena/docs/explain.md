### 3.2 소스: `qa/replay_source.py`의 `iter_samples_from_jsonl(path)`
`[갱신 2026-07-23]` 깨진 줄에서 예외를 던지던 것을 건너뛰도록 고쳤다. 텔레메트리와 세션 사본은 기록 도중에 세션이 끝나면 마지막 줄이 절반만 쓰인 상태로 남는 것이 정상인데, `TailSource`는 이를 견디고 이쪽만 죽어 같은 데이터를 오프라인 경로에서 읽지 못했다. 건너뛴 줄은 `on_skip` 콜백으로 알리고 도구가 화면에 표시한다.
- 입력: 텔레메트리 파일 경로 `path`(문자열).
- 처리 순서:
  1. `open(path, ...)`로 파일을 UTF-8 텍스트 모드로 연다.
  2. 파일을 한 줄씩 순회한다. 한 줄이 한 틱이다.
  3. `line.strip()`으로 공백/개행을 제거하고, 빈 줄은 건너뛴다.
  4. `json.loads(line)`으로 JSON 문자열을 파이썬 딕셔너리 `record`로 변환한다.
  5. `StateSample(**record)`로 딕셔너리를 키워드 인자로 풀어 `StateSample` 객체를 만든다.
- 반환: `yield`로 `StateSample`을 하나씩 흘려보내는 **제너레이터**다. 전체를 메모리에 올리지 않고 한 번에 하나씩 처리한다.
- 넘기는 곳: 이 스트림은 `run_invariants.py`의 `main`이 `for` 루프로 받는다.

### 3.4 검출: `qa/invariants.py`의 `InvariantChecker.check(sample)`
- 입력: `StateSample` 하나.
- 처리 순서:
  1. 이번 틱의 결과를 담을 빈 리스트 `bugs`를 만든다.
  2. 무상태 규칙 네 개를 차례로 호출한다: `_check_floor`, `_check_bounds`, `_check_health`, `_check_speed`. 각 규칙은 위반 시 `Bug`를, 아니면 `None`을 반환한다. `None`이 아니면 `bugs`에 추가한다.
  3. 상태가 필요한 규칙 `_check_stuck`를 호출한다. 이 규칙은 내부 이력을 갱신하며, 위반 시 `Bug`를 반환한다.
  4. `bugs`를 반환한다. 위반이 없으면 빈 리스트다.
- 반환: `list[Bug]`.
- 받는 곳: `run_invariants.py`의 `main`(3.3)이 받는다.

#### 각 규칙의 판정 기준

### 3.5-2 이상탐지 오라클: `qa/anomaly.py`

### 3.5-3 워치독 오라클: `qa/watchdog.py`
- **관측 대상이 ①·②와 다르다.** 앞의 둘은 봇의 상태를 보지만 이쪽은 게임 프로세스 자체를 본다. 봇이 정상으로 보여도 게임이 버벅이거나 멈춰 있으면 그것도 버그다.
- **성능 측정(게임 수정 불필요)**: 서버는 폴링마다 실제 시각을 알고, 텔레메트리에서 시뮬레이션 시간이 얼마나 흘렀는지도 안다. `실효 틱 = (Δ시뮬 시간 / 프레임 주기) / Δ실제 시간`. 게임이 실시간으로 도는 한 둘은 같은 속도로 흘러야 하므로, 이 비율이 떨어지면 게임이 느려진 것이다.
- **한계**: 이것은 서버 틱이지 클라이언트 렌더 FPS 가 아니다. 화면 프레임은 텔레메트리에 없고 재려면 cgame 계측이 필요하다.
- **CPU·메모리**: psutil 로 게임 프로세스를 찾아 읽는다. 없거나 못 찾으면 그 부분만 꺼지고 틱 측정은 계속된다. 실행 파일 이름은 `QA_PROCESS_HINT` 환경변수로 바꿀 수 있다.
- **진행 없음과 종료의 구분**: 프로세스 상태를 함께 보면 갈린다. 멈춤 + 프로세스 살아 있음 = **진행 없음**. 멈춤 + 프로세스 없음 = 종료(크래시인지 정상인지는 사람이 판단).
- **한계**: "진행 없음"은 진짜 행과 메뉴·점수판 구간을 구분하지 못한다. 셋 다 같아 보인다. 구분하려면 게임의 현재 상태를 텔레메트리에 실어 보내야 한다(QVM 재빌드).
- **밀린 기록과 실시간을 구분한다**: 서버 시작 시 기존 파일을 다 읽는 것은 지난 매치의 기록이지 지금 관측이 아니다. 밀린 기록은 비율이 1을 훨씬 넘으므로(1초에 수백 초치를 읽음) 그것으로 구분한다. **실시간 관측이 확인되기 전에는 어떤 판정도 하지 않는다.** 상태를 `idle` / `catching_up` / `live` 로 나눈다.
- **맵 로딩 중 멈춤은 억제한다.** 파일 잘림 감지를 활용해 로딩 직후 일정 시간 판정을 쉰다.
- **세션과 수명이 다르다.** 맵이 바뀌어도 같은 게임 프로세스가 도므로 서버 전체에 하나만 둔다.
- **성능 임계값은 실측으로 정했다**(2026-07-23). 정상 플레이 71개 표본에서 최소 0.930, 중앙값 0.995, 표준편차 0.419(틱 기준)였다. 임계값은 0.85다. 틱이 5초 평균이라 짧은 멈칫이 희석되므로(0.75초 멈춤 → 0.85), 이보다 낮게 잡으면 짧은 멈칫을 놓친다. 멈춤 판정 시간(5초)은 아직 근거가 없다.

### 3.6 실시간 소스: `qa/tail_source.py`의 `TailSource.poll()`
- 역할: 텔레메트리 파일의 **끝을 따라가며 새로 붙은 줄만** 읽는다. `replay_source.py`가 "파일을 처음부터 끝까지 한 번 읽는 소스"라면 이쪽은 "어디까지 읽었는지 기억했다가 그 뒤부터만 읽는 소스"다. 용도가 달라 파일을 나눴다.
- 반환: `PollResult(samples, restarted, raw)`.
  - `samples`: 이번에 새로 파싱한 `StateSample` 목록. 제너레이터가 아니라 목록인 이유는 한 번에 읽는 양이 폴링 간격만큼(1초에 수십 줄)으로 제한되어 부담이 없고, 호출한 쪽이 "이번에 몇 줄 들어왔는지"를 바로 알아야 하기 때문이다.
  - `restarted`: 파일이 초기화됐는지. 참이면 호출한 쪽은 검출기·집계기·기록기를 모두 새로 만들어야 한다.
  - `raw`: 이번에 읽은 **원본 바이트**. 세션 사본을 만들기 위해 넘긴다. 이 모듈이 직접 쓰지 않는 이유는 읽는 책임과 쓰는 책임을 섞지 않기 위함이고, 그 바이트가 어느 세션의 것인지는 세션을 관리하는 서버만 판단할 수 있기 때문이다.
- 반드시 처리해야 하는 두 가지:
  1. **반쯤 쓰인 줄**: 게임이 계속 덧붙이는 중이라 마지막 줄이 절반만 쓰여 있을 수 있다. 마지막 개행까지만 소비하고 남은 조각은 버퍼에 두었다가 다음 읽기에서 앞에 이어 붙인다. 이 처리를 빠뜨리면 파싱 예외로 요청이 실패한다.
  2. **게임 재시작**: 파일 크기가 기억하던 오프셋보다 작아지면 초기화로 보고 처음부터 다시 읽으며 `restarted=True`를 돌려준다. 옛 끼임 이력이 새 매치로 넘어가면 안 되기 때문이다.
- **바이트 모드로 읽는 이유**: 텍스트 모드에서는 여러 바이트로 이뤄진 문자가 중간에 잘렸을 때 디코딩이 실패한다. 바이트 버퍼에 담고 완결된 줄만 디코딩하면 그 문제가 사라진다.
- 깨진 줄은 건너뛰고 `skipped`로 센다. 한 줄 때문에 감시 전체가 멈추면 안 되지만, 조용히 묻히면 안 되므로 API로 노출한다.
- `[추가 2026-07-23]` `pending` 속성은 아직 개행을 만나지 못한 줄 조각을 돌려준다. 세션을 바꿀 때 새 사본의 맨 앞에 넣기 위한 것이다. 이 조각은 이전 세션 사본에 이미 들어가 있고 다음 읽기는 그 줄 중간부터 시작하므로, 넣지 않으면 새 사본의 첫 줄이 조각난 채로 남아 오프라인 도구가 읽지 못한다.

**세션 교체 시 사본 처리 (2026-07-23 수정)**: 세션이 바뀌는 세 경우의 처리가 서로 다르다. 수동 내보내기는 보류 조각을 새 사본에 넣어야 하고(읽기 위치가 유지되므로), 수동 리셋은 `reset()`으로 버퍼가 비어 넣을 것이 없으며, 게임 재시작 감지는 `poll()` 안에서 이미 버퍼를 비우고 새 파일을 읽었으므로 그 조각이 방금 읽은 `raw`에 들어 있어 넣으면 중복된다. 서버의 `_start_session(seed_pending)`이 이 차이를 담는다.

### 3.7 집계: `qa/aggregate.py`의 `BugAggregator`
- 역할: 프레임 단위 탐지를 **버그 사건** 단위로 묶는다. 봇이 5초간 바닥 아래로 떨어지면 같은 탐지가 백여 건 나오는데, 사람이 검토할 단위는 그 백여 건이 아니라 "봇 3이 12:04부터 5.1초간 바닥을 뚫고 떨어졌다"는 사건 하나다.
- **묶는 기준은 `(봇, 규칙)`이다.** 위치는 넣지 않는다. 낙하 중에는 좌표가 계속 변하므로 위치를 기준에 넣으면 한 낙하가 수십 개로 쪼개진다. 심각도는 규칙에서 파생되므로 넣을 필요가 없다.
- **끊는 기준은 시간 간격 `gap_seconds`(기본 0.5초)다.** 같은 `(봇, 규칙)` 탐지가 이 간격 이내로 이어지면 한 사건, 벌어지면 새 사건이다. 근거: 텔레메트리는 서버 프레임(20Hz)마다 기록되므로 연속 구간의 간격은 0.05초다. 0.5초는 약 10프레임이라, 경계선 위에서 탐지가 한두 프레임 깜빡이는 경우는 묶고 실제로 따로 일어난 사건은 가른다. 이 값은 관측에서 역산한 것이 아니라 정한 상수이며, 실측 프레임 간격의 배수로 잡는 편이 더 정확하다.
- 사건(`BugEvent`)이 담는 것: 봇·규칙·심각도, 시작/끝 시각, 지속 시간, 프레임 수(`hits`), **시작 순간의 상태값**, 규칙별 **극값**과 그 순간의 상태값, 진행 중 여부.
  - 주 증거를 시작 순간으로 둔 이유는 "무엇이 잘못되기 시작했는가"가 진단의 출발점이기 때문이다. 극값은 심각도를 가늠하는 보조 정보다.
- **극값은 규칙별 표(`PEAK_SPEC`)로 처리한다.** 각 규칙이 이미 `Bug.details`에 자기가 위반한 수치를 넣고 있으므로, 집계기는 규칙의 내용을 모른 채 "규칙 이름 → (details 키, 최대/최소)" 표만 보고 추적한다. 바닥 관통은 최저 z, 경계 이탈은 최대 `overshoot`, 체력은 최대 체력, 과속은 최대 속력, 끼임은 최대 정지 시간이다. 표에 없는 규칙이 와도 극값만 비고 나머지 집계는 정상 동작한다.
- **상태 기계로 만든 이유**: 파일 전체를 받아 한 번에 묶는 함수로 만들면 실시간으로 바꿀 때 다시 써야 한다. 탐지를 하나씩 받는 구조라, 지금은 파일 전체를 먹이고 실시간에서는 새로 읽은 것만 먹인다. 집계기 코드는 두 경우에 동일하다.
- `finalize(stream_time)`: 마지막 관측 시각을 기준으로, 간격이 벌어진 사건은 닫고 이어지던 사건은 진행 중으로 표시한다. **정적 모드에서 '진행 중'은 "파일이 사건 도중에 끝났다", 실시간에서는 "지금 벌어지고 있다"는 뜻이다.** 같은 코드가 두 의미를 모두 표현한다.
- `on_close` 콜백: 사건이 닫힐 때 호출한다. 서버가 여기에 파일 기록기를 연결한다. **집계기는 파일을 모른다.** 검출기가 집계기를 모르는 것과 같은 이유로 의존 방향을 한쪽으로만 둔다.
- **층의 구분은 폴더가 아니라 호출 방향으로 정한다.** 이 모듈은 `qa` 패키지 안에 있지만 검출기의 일부가 아니다. 검출기는 집계를 부르지 않고, 소비자(서버·CLI)가 검출기 뒤에 집계를 붙인다.

### 3.8 기록: `qa/session_log.py`의 `SessionLog`
- 역할: 한 세션 동안의 사건과 텔레메트리 원본을 파일로 남긴다. 세션 폴더는 기본 `sessions/`이며, 파일 세 종류가 만들어진다.
  - `session_<시각>.jsonl` — 사건 목록(한 줄에 하나)
  - `session_<시각>.summary.json` — 요약과 **그때 사용한 검출 설정값**
  - `session_<시각>.telemetry.jsonl` — 텔레메트리 원본 사본
- **사건을 세션 종료 시 일괄 덤프하지 않고, 닫힐 때마다 한 줄씩 덧붙인다.** 프로세스가 언제 죽어도 그때까지 닫힌 사건은 디스크에 남고 손실은 마지막 한 줄 이하다. 배열 하나로 감싼 JSON이면 닫는 괄호를 못 써서 파일 전체가 깨진다. 이 프로젝트가 잡으려는 대상 중에 크래시와 행이 있는데 기록 시스템이 크래시에 다 날아가면 앞뒤가 맞지 않는다.
- **요약에 검출 설정값을 넣는 이유는 재현성이다.** "이 결과가 어떤 경계값과 임계값으로 나왔는가"를 남기지 않으면 나중에 세션 파일만 봤을 때 판정의 근거를 재구성할 수 없다.
- **텔레메트리 사본을 읽는 즉시 만드는 이유**: 게임은 맵 로드마다 텔레메트리를 `FS_WRITE`로 지운다. 맵 전환은 "파일이 작아졌다"로 **사후에** 감지하므로, 그 시점에 복사하려 하면 원본은 이미 없다. 그래서 서버가 새 바이트를 읽을 때마다 그 바이트를 사본에 덧붙인다. 어차피 읽고 있는 내용이라 추가 비용이 거의 없다.
- **사본은 가공하지 않는다.** 절반만 쓰인 줄이 섞여 있어도 손대지 않으며, 나머지는 다음 호출에서 이어 붙는다. 결과적으로 사본은 원본과 바이트 단위로 동일하다. 파싱해 다시 쓰면 숫자 표기가 미묘하게 달라져 사본이 원본이 아니게 된다.
- flush 정책이 둘로 나뉜다. 사건은 드물어 **매번 flush** 하고, 텔레메트리는 초당 수십 KB가 흘러 매번 flush 하면 낭비이므로 **닫을 때 비워지는 것에 맡긴다.** 대신 프로세스가 갑자기 죽으면 사본 끝의 몇 KB가 유실될 수 있다.

### 3.9 서버: `dashboard/server/app.py`
- `LiveSession`이 한 세션의 감시 상태(검출기·집계기·기록기·최근 원시 탐지·처리 줄 수)를 한데 들고 있다. 검출기와 집계기 모두 내부 상태를 가지므로, 함께 만들고 함께 버릴 수 있도록 한 객체에 모았다. 게임이 재시작되면 통째로 새로 만든다.
- `_poll()`의 **순서가 중요하다**: 읽기 → (초기화됐으면) 세션 교체 → 사본 쓰기 → 검출·집계. 초기화된 폴링에서 읽은 바이트는 이미 새 매치의 것이므로, 사본을 먼저 쓰면 이전 매치 사본에 섞인다.
- 상태를 만지는 구간은 `threading.Lock`으로 감싼다. FastAPI 의 동기 엔드포인트는 스레드풀에서 실행되므로 폴링이 잦아지면 요청이 겹칠 수 있고, 그러면 읽기 위치가 어긋나거나 같은 줄을 두 번 먹는다.
- **무엇을 보관하고 무엇을 버리는가**: 사건은 전부 보관하고 파일로도 남긴다. 시스템이 내린 결론이기 때문이다. 원시 탐지는 최근 500건만 링 버퍼로 유지하고 누적 개수만 센다. 텔레메트리만 있으면 언제든 다시 만들 수 있는 파생물이기 때문이다. **재현 가능한 파생물은 버려도 되고, 결론은 버리면 안 된다**는 것이 기준이다.
- 엔드포인트
  - `GET /api/events?limit=300` — 집계된 사건 목록. 대시보드가 쓴다. `limit`은 오가는 양을 줄일 뿐이며 보관량과 무관하다. `[추가 2026-07-23]` 응답에 이상탐지 결과가 `anomaly` 블록으로 함께 실린다. 사건 목록과 섞지 않는 이유는 하드 인바리언트가 확정이고 이상탐지는 단서이기 때문이다.
  - `POST /api/reload_model` `[추가 2026-07-23]` — 이상탐지 모델을 디스크에서 다시 읽는다. 서버를 켜 둔 채 새로 학습했을 때 쓴다. 서버 재시작은 감시 상태를 통째로 날리므로 모델만 바꾸려고 재시작하는 것은 손해가 크다.
  - `GET /api/detections` — 최근 원시 탐지. 집계가 무엇을 어떻게 묶었는지 대조하는 창구다. 대조할 수 없으면 집계 결과를 신뢰할 근거가 없다.
  - `GET /api/session` — 현재 세션의 파일 경로와 기록 상태.
  - `POST /api/export` — 진행 중인 사건까지 확정 기록하고 요약을 쓴다.
  - `POST /api/reset` — 세션을 마감하고 텔레메트리를 처음부터 다시 읽는 새 세션을 시작한다. 파일이 잘리지 않고 이어 쓰이는 경우처럼 자동 감지가 안 되는 상황에서 쓴다.
- 환경변수: `QA_TELEMETRY`(감시할 파일), `QA_SESSION_DIR`(세션 폴더), `QA_SESSION_LOG=0`(파일 기록 끄기), `QA_ARCHIVE_TELEMETRY=0`(사본 끄기).
- **`--reload`를 쓰면 안 된다.** 파일이 바뀔 때마다 서버가 재시작되면서 감시 상태가 통째로 날아간다.

### 3.10 표현: `dashboard/frontend/src/qa_dashboard.jsx`
- 1초 간격으로 `/api/events`를 폴링한다. `setInterval`이 아니라 매번 `setTimeout`을 새로 거는 방식을 쓴다. `setInterval`은 응답이 늦어도 다음 요청을 그대로 쏘아 요청이 쌓이는데, 이 방식은 한 요청이 끝난 뒤에야 다음 타이머를 걸므로 겹치지 않는다.
- 실패를 두 가지로 나눈다. **연결 자체가 안 되면**(서버 미기동) 3초 간격으로 자동 재시도하고, **서버가 오류 응답을 주면** 자동 재시도하지 않는다. 서버는 살아 있고 안에서 뭔가 잘못됐다는 뜻이라 계속 찔러도 상황이 안 바뀌고 문제만 가려진다.
- 탭이 가려지면(`visibilitychange`) 폴링을 멈춘다.
- **상세 패널의 선택은 사건 객체가 아니라 사건 번호로 들고 있다.** 1초마다 목록을 새로 받으므로 객체를 붙들면 참조가 끊겨 패널이 닫힌다. 번호로 들면 진행 중 사건의 지속 시간과 프레임 수가 패널에서도 실시간으로 갱신된다.
- 서버 응답을 `normalizeEvent()`로 한 번 변환해 쓴다. 서버 응답은 화면 전용 포맷이 아니라 API 계약이므로 표기 편의로 이름을 줄이지 않고 여기서 흡수한다. 이상탐지 오라클이 같은 엔드포인트에 섞여 들어와도 이 함수만 손보면 된다.
- LIVE 표시는 정적 시절에 뺐다가 실시간이 된 뒤 되살렸다. **표시의 정당성은 표시 자체가 아니라 그 뒤의 동작에서 나온다.** 멈춰 있는 데이터 위의 LIVE 램프는 사실과 어긋난다.
- 화면에 없는 값은 지어내지 않는다. 봇 이름과 맵 이름은 텔레메트리에 없으므로 엔티티 번호로만 표시한다.
- `[추가 2026-07-23]` 이상탐지 결과는 좌측 `이상 구간` 패널에 별도로 표시한다. 상태를 네 가지로 구분한다(모델 없음 / 첫 창 채우는 중 / 이상 없음 / 목록). 창이 60초라 세션 시작 후 1분간 결과가 없는 것이 정상인데, 그것이 정상임을 모르면 고장으로 보이기 때문이다. 항목마다 어떤 특징이 몇 표준편차 벗어났는지를 함께 보여 주고, 클릭하면 상세 패널에 특징별 벗어난 정도가 막대로 표시된다.
- 상세 패널의 선택은 `{kind, id}` 형태다. 사건과 이상 항목이 서로 다른 목록에서 오고 번호도 각자 매기므로 번호만으로는 구분할 수 없다.

    python tools/measure_interval.py sessions/session_<시각>.telemetry.jsonl

같은 봇의 연속된 두 줄 사이 `time` 차이를 모아 서버 프레임 주기를 구하고, 그보다 훨씬 큰
공백(봇 사망 구간 등)을 찾아 준다. 집계의 `gap_seconds`를 정할 때 쓴다. 공백이
`gap_seconds`보다 크면 하나의 버그 사건이 둘로 갈라지므로, 공백의 크기가 실질적인 기준이 된다.

### 4.4 맵 경계 보정

    python tools/calibrate_bounds.py <정상 플레이 텔레메트리>

출력된 `MapBounds`와 `max_speed`를 `run_invariants.py`와 `dashboard/server/app.py` **양쪽에** 반영한다. 주입을 끈 상태의 텔레메트리로 보정해야 한다.
