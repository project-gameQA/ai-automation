## 1. 프로젝트 의의
- 목적: OpenArena를 대상으로 게임 QA 자동화의 실제 예시를 구현하고, 발표로 시연 가능한 형태로 보여준다.
- 핵심 주장: 봇이 게임을 플레이하는 동안 시스템이 자동으로 버그를 탐지하고, 그 결과를 사람이 검토할 수 있도록 누적·시각화한다.
- OpenArena를 대상으로 선택한 이유는 다음과 같다.
  - 오픈소스 C++라 소스 계측과 버그 주입이 가능하다.
  - 봇이 내장돼 있어 정상 플레이 데이터를 확보하기 쉽다.
  - 콘솔·데모·timedemo 등 자동화에 활용할 자원이 갖춰져 있다.
  - 저사양 하드웨어(Intel HD 4600)에서도 구동된다.

## 2. 핵심 설계 원칙: 액터와 오라클의 분리
- **액터(actor)**: 게임 안에서 실제로 움직이며 상태를 생성하는 주체다. (내장 봇 / 스크립트 봇 / 강화학습 에이전트)
- **오라클(oracle)**: 특정 상태가 버그인지 판정하는 검출기다.
- 이 둘은 독립적인 축이다. 강화학습은 액터 선택지일 뿐, 그 자체로 버그를 판정하지 않는다.
- 따라서 "정상 플레이 학습 기반 이상탐지" 대 "강화학습"은 양자택일 관계가 아니다. 어떤 액터를 쓰든 오라클은 별도로 필요하다.
- 강화학습이 QA에 기여하는 지점은 오직 **탐색 범위(coverage)** 확대다. 다만 이기도록 학습된 에이전트는 버그 상태를 회피하므로, 버그를 찾게 하려면 보상을 "새로운/희귀한 상태 도달"로 설계해야 한다.

## 3. 아키텍처
전체 흐름은 다음과 같다.

    액터 → 수집 → 오라클(3계층) → 집계 → 대시보드(사람 검토)

### 3.1 액터 층
- OpenArena 게임 엔진과 봇으로 구성한다.
- 평가와 시연을 위해 재현 가능한 합성 버그를 소스에 주입한다.

### 3.2 수집 층
- 텔레메트리: 봇의 위치·속도·체력·시야각·탄약·이벤트를 매 틱 수집한다.
- 로그 / 이벤트 스트림을 수집한다.
- 프로세스 지표: CPU·메모리·생존 여부를 수집한다.

### 3.3 오라클 층 (3계층)
- **① 하드 인바리언트(결정적 규칙)**: Z좌표가 맵 바닥 아래, 체력 범위 초과, 좌표 경계 밖(OOB), 일정 시간 위치 불변(끼임) 등을 판정한다. 오탐이 거의 없고 설명이 명확하다.
- **② 이상탐지(비지도 ML)**: 정상 봇 플레이로 학습하여 편차를 탐지한다. 규칙은 어기지 않지만 이상한 행동을 잡는다. 오탐이 있어 사람 검토가 필요하다.
- **③ 크래시/프로세스 워치독**: 크래시·행(멈춤)·메모리 증가를 게임 밖에서 감시한다.

### 3.4 집계 층
- 세 오라클의 출력을 모아 버그 목록으로 누적 저장한다.

### 3.5 표현 층
- 라이브 대시보드에 게임 플레이 화면, 실시간 이벤트 로그, 누적 버그 테이블(타임스탬프·유형·심각도·스크린샷)을 표시한다.
- 사람이 대시보드를 검토한다(human-in-the-loop).

## 4. 결정 로그
- 2026-07-20: 대상 게임을 OpenArena로 확정한다.
- 2026-07-20: 오라클을 3계층(하드 인바리언트 + 이상탐지 + 크래시/프로세스 감시)으로 구성하기로 한다.
- 2026-07-20: v1에서는 강화학습을 채택하지 않고 보류한다. 사유는 다음과 같다.
  - (1) 강화학습은 액터일 뿐 버그 판정기가 아니다.
  - (2) 이기도록 학습된 에이전트는 버그 상태를 회피하므로 테스터로 부적합하다.
  - (3) OpenArena에는 즉시 쓸 수 있는 RL 인터페이스가 없고, 학습 비용이 크며, 현재 하드웨어에서 비현실적이다.
  - 강화학습은 향후 "탐색 커버리지 확장"으로 후순위에서 재검토한다.
- 2026-07-20: 합성 버그 주입 방식을 "cvar로 토글하는 결함 주입기를 게임 모듈(qagame) 코드에 심는 방식"으로 확정한다. 세부 방침은 다음과 같다.
  - 엔진이 아니라 게임 모듈만 수정·빌드하며, 결함 주입기는 텔레메트리 계측 훅과 같은 층에 둔다.
  - `qa_inject_*` cvar로 각 버그 유형을 개별 토글해 발표 때 정해진 시점에 재현 가능하게 한다.
  - 내가 언제 무엇을 켰는지 알 수 있으므로, 주입한 버그는 정답이 붙은 평가 데이터로 활용한다.
  - 빠른 데모용으로 기존 cvar(예: g_gravity, g_speed)를 보조 수단으로 쓴다.
- 2026-07-20: 대상 게임 소스를 `OpenArena/gamecode`(OAX)로 확정하고, 게임 로직을 네이티브 DLL이 아니라 **QVM 모드**로 빌드하기로 한다. 사유는 QVM이 아키텍처 독립적이라 스톡 0.8.8 엔진을 그대로 쓸 수 있고, 동봉된 lcc/q3asm 도구로 빌드해 최신 gcc와 옛 코드의 충돌 문제도 피하기 때문이다. (상세: docs/injection_map.md 5-B)
- 2026-07-20: 결함 주입과 텔레메트리 계측을 모두 `g_active.c`의 `ClientEndFrame`, 체력 동기화(`ps.stats[STAT_HEALTH] = ent->health`) 직후에 둔다. 그 자리가 상태가 프레임마다 확정되는 지점이라, 주입값이 최종 상태로 살아남고 계측도 그 확정값을 읽을 수 있기 때문이다.
- 2026-07-20: 텔레메트리는 QVM 제약상 직접 파일 입출력을 못 하므로 엔진 syscall(`trap_FS_FOpenFile`/`trap_FS_Write`/`trap_FS_FCloseFile`)로 파일에 쓴다. 형식은 `StateSample` 스키마와 동일한 JSONL로 하여, 기존 `replay_source`/`run_invariants`가 그대로 소비할 수 있게 한다. 봇(`SVF_BOT`)만 기록한다.
- 2026-07-20: 체력 하드 인바리언트를 실제 게임 규칙에 맞게 교정한다. 배경과 사유는 다음과 같다(실제 텔레메트리 검증 중 발견한 오탐).
  - 발견: `qa_inject_health`가 꺼진 정상 플레이에서도 `health_out_of_range`가 잡혔다. 예: 체력 95, max_health 70.
  - 원인: 검출기 규칙이 게임의 실제 규칙과 어긋났다. OpenArena에서는 체력이 max_health를 넘는 것이 정상이다. `g_client.c`에서 스폰 시 체력을 `max_health + 25`로 주고(70이면 95로 시작), `g_items.c` 주석대로 소형/메가헬스는 "will go over the max"이며, 가드 파워업은 최대 `2*handicap`(핸디캡 100이면 200)까지 올린다. 즉 정당한 체력 상한은 약 200이다.
  - 교훈: 하드 인바리언트는 "드물다"가 아니라 "물리적으로 불가능하다"를 담아야 한다. `health > max_health`는 정상 오버힐을 오탐하므로 부적합했다. "최대치를 약간 넘는" 애매한 영역은 하드 규칙이 아니라 이상탐지(ML)의 몫이다.
  - 수정: 규칙을 '절대 상한 초과'로 변경했다(`health > max_possible_health`, 기본 250). 정당한 상한(~200)보다 넉넉히 위, 주입 오버플로우(999)보다 아래로 잡아 정상 오버힐은 통과시키고 진짜 오버플로우만 잡는다. 하한(`health < 0`)도 사망 시 음수가 정상이라 오탐 소지가 있어 제거했다.
  - 검증: 실제 텔레메트리에서 정상 구간의 체력 95 등이 더 이상 잡히지 않고, 주입값 999만 잡히는 것을 확인했다.
- 2026-07-20: 맵 경계(`MapBounds`)는 게임 내부 데이터가 아니라 **정상 플레이 관측**에서 역산하기로 한다(`tools/calibrate_bounds.py`). 근거는 다음과 같다.
  - 게임(`.bsp`)도 맵 경계 데이터(worldmodel의 mins/maxs)를 갖지만, 그것은 걸어다닐 수 있는 영역이 아니라 맵 기하 전체(벽 바깥·스카이박스 포함)를 감싸는 큰 상자다. 이걸 기준으로 삼으면 봇이 벽을 뚫고 나가도 그 큰 상자 안이라 경계 이탈을 놓친다. QA가 원하는 것은 "기하의 끝"이 아니라 "봇이 정상적으로 있을 수 있는 영역의 끝"이며, 둘은 다르다.
  - game-agnostic 원칙과의 일관성. 경계를 게임 내부 API로 뽑으면 맵마다 게임 내부에 결합되어 원칙이 흐려진다. 관측 방식은 텔레메트리만으로 경계를 정한다.
  - 한계: 관측 방식은 봇이 아직 안 가본 정상 영역을 경계 밖으로 오판할 수 있으므로, 충분히 오래 수집하거나 여유(margin, 현재 20%)를 둔다. 필요 시 `.bsp` 경계를 절대 한계로, 관측 경계를 정상 범위로 쓰는 이중 판정으로 확장할 수 있다.
- 2026-07-20: 대시보드 방식을 확정했다.
  - 프론트엔드는 리액트(Vite)로, 백엔드는 파이썬 FastAPI로 한다. 리액트는 브라우저에서 도는 클라이언트(화면)이고, 파이썬 서버가 데이터(검출 결과)를 준다. 전부 localhost 전용이라 인터넷과 무관하게 동작한다.
  - 게임 화면은 대시보드 안에 넣지 않는다. 브라우저에 게임 영상을 넣으려면 캡처·스트리밍이 필요해 복잡하고, 녹화 영상을 로그와 시점 맞춰 트는 방식은 "실제 탐지"가 아니라 짜맞춤처럼 보여 정직하지 않다. 대신 게임을 **별도 창**으로 대시보드 옆에 띄운다. 두 창이 각각 실제로 돌고 그 사이를 텔레메트리가 실제로 흐르므로 정직하고 구현도 단순하다. 최종 시연은 두 창을 함께 화면 녹화한다.
  - 데이터 전달은 파일 tail 방식으로 시작한다(파이썬이 파일 끝을 주기적으로 읽고 프론트엔드가 폴링). 0.1~0.5초 지연은 탐지가 영상이 아니라 사건이라 문제되지 않는다. 실시간 최적화(WebSocket 푸시)는 나중 선택지로 남긴다.
  - 화면은 3열 구조로 한다. 좌: 규칙별 집계 + 이상탐지 자리, 중앙: 실시간 탐지 로그, 우: 클릭 시 열리는 상세 보기 패널. 의미 없는 통계(총계·심각·주의·분당 탐지)는 프레임 단위 중복 때문에 실제 버그 수와 무관해 오해를 부르므로 넣지 않는다. 상세 보기를 인라인 확장이 아니라 옆 패널로 둔 것은, 이상탐지 상세(점수·기여 특징·그래프)가 세로로 길어 실시간 로그 흐름을 방해하지 않게 하기 위해서다.
  - 집계(프레임 중복을 하나의 버그 사건으로 묶기)는 검출기가 아니라 서버/대시보드 층에서 한다. 검출기는 매 프레임 판정만 하고, 중복 묶기는 표현의 문제이기 때문이다.
- 2026-07-20: 하드 인바리언트의 맵 의존성은 알려진 한계로 둔다.
  - 다섯 규칙 중 경계·바닥·과속은 맵마다 다른 값(경계·속도 상한)에 의존하므로 맵이 바뀌면 재보정이 필요하다. 체력·끼임은 맵과 무관하다.
  - 포트폴리오에서는 한 맵(예: q3dm17) 데모로 충분하므로 지금 다맵 지원은 하지 않는다.
  - 확장 방향: (1) 맵별 설정 파일에 보정값을 저장해 재사용, (2) 텔레메트리에 현재 맵 이름 필드를 추가해 서버가 자동으로 맞는 설정을 선택, (3) 관측 기반 자동 학습으로 맵 의존성을 데이터로 흡수(이상탐지와 연결). 지금은 한계·확장 방향으로만 남긴다.

- 코드 수정이 게임에 반영되는 루프를 검증했다(`G_InitGame`에 로그 한 줄 추가로 확인).
- 첫 결함 주입기 `qa_inject_health`를 구현하고, 콘솔에서 토글 시 봇/플레이어 체력이 999로 고정되는 것을 확인했다.
- 텔레메트리 훅을 구현했다. `ClientEndFrame`에서 봇 상태를 JSONL로 파일에 기록하며, 주입값이 그대로 실려 나가도록 주입 직후에 배치했다.
- 엔드 투 엔드 흐름을 실제 게임에서 검증했다. 봇 매치에서 `qa_inject_health`를 토글해 얻은 실제 텔레메트리를 `run_invariants.py`에 넣어, 주입 구간의 999가 `health_out_of_range`로 검출되는 것을 확인했다. 게임 → 텔레메트리 → 검출기 → 판정의 첫 수직 흐름이 완성됐다.
- 실제 데이터로 체력 규칙의 오탐을 발견하고 교정했다(결정 로그 참조). 정상 오버힐(체력 95 등)이 더 이상 잡히지 않고 주입값 999만 잡히는 것을 확인했다.
- 맵 경계를 `tools/calibrate_bounds.py`로 정상 플레이 텔레메트리에서 산출해 `run_invariants.py`에 반영했다. 그 결과 `out_of_bounds` 오탐이 정상 구간에서 0건이 됐다(음성 검증 통과).
- 경계 이탈 주입기 `qa_inject_oob`를 추가하고, 봇을 x=100000으로 강제했을 때 `out_of_bounds`가 잡히는 양성 검증을 확인했다.
- 바닥 관통 주입기 `qa_inject_fall`를 콜리전 제거(tracemask에서 `CONTENTS_SOLID`·`CONTENTS_BOTCLIP` 제거) 방식으로 구현했다. 봇이 중력 가속대로 실제 추락하며 z가 내려갔고, `fell_through_floor`가 잡혔다. 이 방식은 화면 시연(자연 낙하)과 검출을 동시에 만족한다. 낙하 속도가 커지며 `impossible_velocity`도 함께 잡혔는데, 이는 자유낙하의 정상 반응이지 오탐이 아니다.
- 끼임 규칙을 "오래 정지"에서 "이동 입력이 있는데 위치가 안 변함"으로 개선했다(텔레메트리에 `move_input` 추가, 검출기에 입력 게이트 적용). 정상 대기(입력 없음)는 더 이상 오탐되지 않아 정상 플레이에서 0건이 됐다.
- 끼임 주입기 `qa_inject_stuck`를 추가했다(이동 입력은 살리고 위치만 고정). 켰을 때 봇들이 `stuck`으로 검출돼 양성 검증을 통과했다.

현재 검증 상태(갱신): **검출기 다섯 규칙(체력·경계·바닥 관통·과속·끼임)이 모두 양성·음성 검증을 통과했다.** 하드 인바리언트 오라클(①)이 실제 게임 데이터로 완결됐다.

표현 층(대시보드) 진행:
- 리액트 대시보드(mock 데이터) 1차 완성. 3열 구조(좌: 규칙별 집계 + 이상탐지 자리 / 중앙: 실시간 탐지 로그 / 우: 상세 보기 옆 패널), 인광 그린 계기판 톤. 브라우저에서 렌더 확인. 데이터 소스 블록을 표시해 나중에 파이썬 API 폴링으로 교체할 자리를 남겼다.
- FastAPI 서버(`dashboard/server/app.py`) 작성. 텔레메트리를 읽어 `InvariantChecker`를 돌리고 `/api/detections`로 버그 목록을 내보낸다. 각 탐지에 그 순간의 텔레메트리 값(좌표·속도·체력 등)을 실어 상세 패널이 바로 쓰게 했다. 정적 버전(파일 전체를 한 번에 검출)으로 시작한다. 컨테이너에서 검출 로직 동작을 확인했다.
- 주의: 서버의 경계·임계값이 `run_invariants.py`와 중복돼 있다. 향후 `qa/config.py` 공용 설정으로 통합한다.
