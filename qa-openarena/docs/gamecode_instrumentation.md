## 1. 개요
게임 소스에 가한 변경은 두 종류다. 하나는 **결함 주입기**(의도한 버그를 재현 가능하게 발생시킴), 다른 하나는 **텔레메트리 계측**(봇 상태를 외부로 내보냄)이다. 둘 다 게임 로직 모듈(qagame)의 같은 층에 들어가며, 엔진은 건드리지 않는다. 빌드는 QVM 모드로 하여 스톡 0.8.8 엔진에 얹는다.

변경한 파일은 세 개다.
- `code/game/g_main.c` — 게임 초기화·종료·프레임 루프·cvar 등록이 있는 서버 게임 로직의 중심.
- `code/game/g_local.h` — 게임 모듈 내부에서 공유하는 선언(전역 변수의 extern 등)을 모은 헤더.
- `code/game/g_active.c` — 클라이언트(봇 포함) 상태를 매 프레임 갱신하는 로직.

## 2. 변경 목록

### 2.1 [g_main.c] 빌드/적재 확인용 로그
- 위치: `G_InitGame` 함수 시작부.
- 이 부분의 역할: `G_InitGame`은 맵이 로드될 때 한 번 호출되어 게임 상태를 초기화한다.
- 넣은 코드: `G_Printf("QA MOD LOADED: build check ok\n");`
- 넣은 이유: 내가 빌드한 qagame QVM이 실제로 로드됐는지 콘솔에서 확인하기 위한 표시다. 이후 주입기가 동작하지 않을 때, 그것이 코드 문제인지 아니면 애초에 내 QVM이 로드조차 안 된 것인지를 구분하는 기준점이 된다. (검증용 임시 로그이며, 안정화 후 제거하거나 디버그 표시로 남긴다.)

### 2.2 [g_main.c] 결함 주입 스위치 cvar 선언
- 위치: 다른 `vmCvar_t g_*;` 전역 선언들이 모인 곳.
- 이 부분의 역할: 게임 모듈이 사용하는 콘솔 변수(cvar)의 값을 담는 전역 변수들을 선언하는 자리다.
- 넣은 코드: `vmCvar_t qa_inject_health;`
- 넣은 이유: 결함 주입을 켜고 끄는 스위치 값을 담을 변수다. cvar는 그 자체로 게임 상태를 바꾸지 않고, "언제 주입할지"를 결정하는 방아쇠 역할만 한다.

### 2.3 [g_local.h] 스위치 cvar의 외부 선언
- 위치: `extern vmCvar_t ...` 선언들이 모인 곳.
- 이 부분의 역할: 한 파일(g_main.c)에서 정의한 전역 변수를 다른 파일에서도 참조할 수 있게 `extern`으로 공유 선언하는 자리다.
- 넣은 코드: `extern vmCvar_t qa_inject_health;`
- 넣은 이유: 스위치 변수는 g_main.c에 정의하지만 실제로 값을 읽어 쓰는 곳은 g_active.c다. C에서는 이렇게 헤더에 extern 선언을 둬야 다른 파일이 같은 변수를 공유한다.

### 2.4 [g_main.c] 스위치 cvar 등록
- 위치: `gameCvarTable[]` 배열 안(`sv_cheats` 항목 부근).
- 이 부분의 역할: 이 테이블에 등록된 cvar를 엔진이 시작 시 등록하고(`G_RegisterCvars`), 매 프레임 콘솔의 최신 값을 해당 변수에 갱신해 넣는다(`G_UpdateCvars`).
- 넣은 코드: `{ &qa_inject_health, "qa_inject_health", "0", 0, 0, qfalse },`
- 넣은 이유: 이 테이블에 넣어야 `qa_inject_health`가 콘솔 명령으로 인식되고, 콘솔에서 `\qa_inject_health 1`로 켠 값이 매 프레임 코드에 반영된다. 기본값 0(꺼짐)으로 두어 평소에는 아무 영향이 없다.

### 2.5 [g_active.c] 체력 결함 주입
- 위치: `ClientEndFrame` 함수 안, `ent->client->ps.stats[STAT_HEALTH] = ent->health;`(체력 동기화) 직후.
- 이 부분의 역할: `ClientEndFrame`은 각 클라이언트(봇 포함)의 상태를 매 프레임 마무리하는 함수다. 문제의 그 줄은 서버측 체력(`ent->health`)을 네트워크·화면에 쓰이는 상태(`ps.stats[STAT_HEALTH]`)로 동기화한다.
- 넣은 코드: `qa_inject_health.integer`가 참이면 `ent->client->ps.stats[STAT_HEALTH] = 999;`로 덮어쓴다.
- 넣은 이유: 체력이 프레임마다 확정되는 지점이 바로 이 동기화 줄이다. 우리 코드를 그 **직후**에 둬야 우리가 덮어쓴 값이 최종 상태로 살아남아 화면과 텔레메트리에 반영된다. 순서가 반대면 정상 동기화가 우리 값을 다시 덮어쓴다.

### 2.6 [g_main.c] 텔레메트리 파일 핸들 선언
- 위치: 2.2의 cvar 선언 부근.
- 이 부분의 역할: 2.2와 같이 게임 모듈의 전역 상태를 선언하는 자리다.
- 넣은 코드: `fileHandle_t qa_telemetryFile;`
- 넣은 이유: 봇 상태를 기록할 파일의 핸들(열린 파일을 가리키는 식별자)을 담는다. 파일을 여는 곳(G_InitGame)과 쓰는 곳(ClientEndFrame)과 닫는 곳(G_ShutdownGame)이 서로 다른 함수/파일이라, 이 핸들을 전역으로 공유해야 한다.

### 2.7 [g_local.h] 텔레메트리 핸들의 외부 선언
- 위치: 2.3의 extern 선언 부근.
- 넣은 코드: `extern fileHandle_t qa_telemetryFile;`
- 넣은 이유: 핸들은 g_main.c에 정의하지만 g_active.c에서도 써야 하므로, 2.3과 같은 이유로 extern 공유 선언이 필요하다.

### 2.8 [g_main.c] 텔레메트리 파일 열기
- 위치: `G_InitGame` 안, 2.1의 로그 직후.
- 이 부분의 역할: 맵 로드 시 한 번 실행되는 초기화 구간이다.
- 넣은 코드: `trap_FS_FOpenFile("qa_telemetry.jsonl", &qa_telemetryFile, FS_WRITE);`
- 넣은 이유: 맵이 시작될 때마다 텔레메트리 파일을 새로 연다(FS_WRITE라 매번 초기화). QVM은 직접 파일 입출력을 할 수 없어, 엔진이 제공하는 파일 syscall을 통해서만 파일을 다룰 수 있으므로 이 함수를 쓴다.

### 2.9 [g_main.c] 텔레메트리 파일 닫기
- 위치: `G_ShutdownGame` 안, 기존 `level.logFile`을 닫는 블록 직후.
- 이 부분의 역할: 게임/맵이 끝날 때 자원을 정리하는 구간이다. 바로 위에 게임 로그 파일을 닫는 기존 코드가 있어, 같은 패턴을 그대로 따른다.
- 넣은 코드: `qa_telemetryFile`이 열려 있으면 `trap_FS_FCloseFile`로 닫고 0으로 초기화한다.
- 넣은 이유: 연 파일은 반드시 닫아야 버퍼가 비워지며 데이터가 온전히 기록된다. 닫지 않으면 마지막 기록이 유실될 수 있다.

### 2.10 [g_active.c] 텔레메트리 기록
- 위치: `ClientEndFrame` 안, 2.5의 체력 주입 직후.
- 이 부분의 역할: 2.5와 같이 각 클라이언트 상태가 프레임마다 확정되는 지점이다.
- 넣은 코드: 엔티티가 봇(`SVF_BOT`)이고 파일이 열려 있으면, 그 봇의 상태(tick, time, entity_id, 위치, 속도, 체력, 최대체력)를 `Com_sprintf`로 JSON 한 줄로 만들어 `trap_FS_Write`로 파일에 쓴다.
- 넣은 이유: 두 가지다. (1) 이 위치가 상태 확정 지점이자 결함 주입 직후라, 주입한 버그값이 그대로 텔레메트리에 실려 나가 검출기가 받을 수 있다. (2) 출력 형식을 검출기의 `StateSample` 스키마와 동일한 JSONL로 맞춰, 기존 Python 검출기가 수정 없이 그대로 소비하게 했다. `SVF_BOT` 검사로 봇만 기록해 파일을 대상(봇) 상태로만 채운다.

### 2.11 [g_main.c / g_local.h] 경계 이탈 주입 스위치 (qa_inject_oob)
- 위치: 2.2~2.4와 동일한 자리(전역 선언, extern, gameCvarTable 등록).
- 넣은 코드: `vmCvar_t qa_inject_oob;` / `extern vmCvar_t qa_inject_oob;` / `{ &qa_inject_oob, "qa_inject_oob", "0", 0, 0, qfalse },`
- 넣은 이유: 경계 이탈 버그를 켜고 끄는 스위치다. 체력 스위치와 완전히 같은 틀을 복제했다.

### 2.12 [g_active.c] 경계 이탈 주입
- 위치: `ClientEndFrame` 안, 2.5의 체력 주입 블록 직후(텔레메트리 기록보다 위).
- 이 부분의 역할: 상태가 프레임마다 확정되는 지점이다.
- 넣은 코드: `qa_inject_oob.integer`가 참이면 `ent->client->ps.origin[0] = 100000;`으로 X 좌표를 맵 밖으로 강제한다.
- 넣은 이유: 검출기의 `_check_bounds`가 x·y·z를 대칭으로 검사하므로, x 하나를 압도적으로 큰 값(경계 x최대보다 훨씬 큼)으로 밀어내면 경계 이탈을 확실히 재현·검증할 수 있다. 텔레메트리보다 위에 둬야 오염된 좌표가 기록된다. 참고: 이는 자연스러운 이동이 아니라 통제된 강제 순간이동이며, "경계 밖 좌표가 텔레메트리에 실리면 검출기가 잡는가"를 검증하는 용도다.

### 2.13 [g_main.c / g_local.h] 바닥 관통 주입 스위치 (qa_inject_fall)
- 위치: 2.2~2.4와 동일한 자리.
- 넣은 코드: `vmCvar_t qa_inject_fall;` / `extern vmCvar_t qa_inject_fall;` / `{ &qa_inject_fall, "qa_inject_fall", "0", 0, 0, qfalse },`
- 넣은 이유: 바닥 관통(콜리전 제거로 추락)을 켜고 끄는 스위치다.

### 2.14 [g_active.c] 바닥 관통 주입 — 콜리전 제거 (새 위치: ClientThink_real)
- 위치: `ClientThink_real` 안, `pm.tracemask`를 설정하는 if/else 블록 직후, `pm.trace = trap_Trace;` 앞. **다른 주입기와 달리 `ClientEndFrame`이 아니라 `ClientThink_real`에 넣는다.**
- 이 부분의 역할: `ClientThink_real`은 이동 계산(`Pmove`)을 준비하고 호출하는 함수다. `pm.tracemask`는 봇이 이번 프레임 이동에서 무엇과 충돌할지를 결정하는 값이다.
- 넣은 코드:
  ```c
  if ( qa_inject_fall.integer ) {
      pm.tracemask &= ~( CONTENTS_SOLID | CONTENTS_BOTCLIP );
  }
  ```
- 넣은 이유(핵심):
  - 왜 이 함수인가: 콜리전은 이동 계산(Pmove)이 실행되기 전에 설정돼야 한다. 그 준비가 `ClientThink_real`에서 이뤄지므로, 상태 확정 지점인 `ClientEndFrame`이 아니라 여기에 넣는다.
  - 왜 이 코드인가: tracemask에서 `CONTENTS_SOLID`(단단한 월드=바닥·벽)를 빼면 봇이 바닥을 통과한다. 이동 타입은 그대로 `PM_NORMAL`이라 중력이 계속 작동하므로, 지면 판정이 사라진 봇은 중력에 의해 실제로 바닥을 뚫고 추락한다. 순간이동이 아니라 자연스러운 낙하다.
  - 왜 `CONTENTS_BOTCLIP`도 빼는가: 맵에는 봇만 막는 보이지 않는 벽(봇 클립)이 바닥 근처에 깔린 경우가 많다. `CONTENTS_SOLID`만 빼면 사람 캐릭터는 떨어지지만 봇은 봇 클립에 걸려 진동만 한다. 봇도 떨어뜨리려면 봇 클립도 함께 제거해야 한다. 우리 텔레메트리는 봇만 기록하므로, 봇이 실제로 떨어져야 검출이 성립한다.
- 검증 결과: 봇들의 z가 중력 가속도대로 −29 → −80 → −300 …으로 가속하며 내려갔고, `fell_through_floor`가 잡혔다. 낙하가 길어지며 속력이 상한을 넘어 `impossible_velocity`도 함께 잡혔는데, 이는 오탐이 아니라 자유낙하 속도가 실제로 빨라진 정상 반응이다.
- 알려진 부작용: 봇이 맵 아래 빈 공간으로 무한히 떨어져 로그가 급증한다. 발표/반복 테스트에서는 일정 깊이 이상 낙하 시 멈추거나 리스폰시키는 다듬기가 필요하다.

### 2.15 [g_active.c] 텔레메트리에 입력 정보 추가 (move_input, attack)
- 위치: 2.10의 텔레메트리 기록 블록(`ClientEndFrame`)을 수정.
- 이 부분의 역할: 봇 상태를 JSONL로 내보내는 계측이다.
- 넣은 코드: 봇의 이번 프레임 입력(`ent->client->pers.cmd`)에서 이동 명령 유무(`forwardmove`/`rightmove`가 0이 아니면 `move_input=1`)와 발사 버튼 유무(`buttons & BUTTON_ATTACK`이면 `attack=1`)를 계산해, JSON에 `move_input`·`attack` 필드로 추가한다.
- 넣은 이유: 끼임 규칙을 "오래 정지"에서 "이동 입력이 있는데 위치가 안 변함"으로 개선하기 위해서다. 봇의 움직이려는 의도(입력)를 검출기가 알아야 정상 대기(입력 없음)와 진짜 끼임(입력 있으나 못 움직임)을 구분할 수 있다. 입력 정보는 `ClientThink_real`이 아니라 `ClientEndFrame`에서도 `pers.cmd`로 접근 가능하다. `attack`은 지금 규칙엔 쓰지 않지만 향후 발사 관련 판정을 위해 미리 넣었다.

### 2.16 [g_main.c / g_local.h / g_active.c] 끼임 주입 (qa_inject_stuck)
- cvar 스위치: 2.2~2.4와 동일한 틀(`vmCvar_t qa_inject_stuck;` / extern / `gameCvarTable` 등록).
- 상태 필드: `g_local.h`의 `clientPersistant_t`에 `vec3_t qaStuckOrigin`(고정 기준 좌표)과 `qboolean qaStuckAnchored`(기준이 잡혔는지)를 추가한다. 클라이언트별로 유지돼야 하므로 `pers`(cmd가 있던 구조체)에 둔다.
- 주입 코드 위치: `ClientEndFrame`, 텔레메트리 기록보다 위. 스위치가 켜지면 첫 프레임에 현재 위치를 기준점으로 저장하고, 이후 매 프레임 위치를 그 기준점으로 되돌려 고정한다(속도도 0으로 하여 물리적 튐 방지). 꺼지면 기준점을 초기화한다.
- 넣은 이유(핵심): 이동 명령(usercmd)은 전혀 건드리지 않으므로 봇 AI는 계속 이동 입력을 내보내 `move_input`이 1로 유지된다. 그런데 위치는 고정되므로, 정확히 "입력은 있는데 결과가 없는" 상태가 만들어진다. 이것이 2.15로 개선한 끼임 규칙의 양성 검증 대상이다. 만약 이동 명령까지 0으로 만들면 `move_input`이 0이 되어 규칙이 정상 대기로 오판하므로, 입력을 살리는 것이 핵심이다.
- 검증 결과: `qa_inject_stuck`을 켰을 때 봇들이 제자리에 고정된 채 이동 입력을 계속 내보냈고, `stuck: 이동 입력이 있는데 2.0초간 위치가 변하지 않아...`로 검출됐다. 정상 플레이(입력 없는 대기)에서는 0건으로, 음성·양성 검증을 모두 통과했다.

### 주입기 순서 규칙 (중요)
- 상태를 오염시키는 모든 주입(체력·경계·끼임 등)은 텔레메트리 기록 블록보다 **위**에 있어야 오염된 값이 파일에 실린다. 특히 위치를 바꾸는 주입(oob, stuck)이 텔레메트리 아래에 있으면 검출이 성립하지 않는다.
- 바닥 관통(콜리전 제거)은 예외적으로 `ClientThink_real`에 있으며, 이는 Pmove 전에 tracemask를 바꿔야 하기 때문이다.

## 2.5절 보충: 비트플래그와 비트마스크
2.14에서 쓴 `pm.tracemask &= ~( CONTENTS_SOLID | CONTENTS_BOTCLIP )`을 이해하려면 비트 연산을 알아야 한다. 게임 코드 곳곳에서 쓰이므로 정리해 둔다.

### 개념
- 정수는 컴퓨터 안에서 이진수로 저장되고, 그 각 자리(bit)를 켜짐(1)/꺼짐(0) 스위치로 쓸 수 있다.
- 이렇게 정수 하나에 여러 개의 예/아니오 설정을 담는 것을 **비트플래그(bit flag)** 또는 **비트마스크(bitmask)**라 한다.
- 각 설정 상수(예: `CONTENTS_SOLID`)는 이진수에서 서로 다른 한 자리만 1인 값으로 정해진다. 그래서 겹치지 않고 합치거나 뺄 수 있다.
- 장점: 메모리 절약(스위치 32개를 정수 하나에)과 속도(여러 스위치를 한 번의 연산으로 처리). 매 프레임 수많은 판정을 하는 게임 코드에 적합하다. `ent->r.svFlags & SVF_BOT`도 같은 원리다.

### 연산자
- `|` (OR): 비트를 켜서 합친다. `0001 | 0100 = 0101`. 두 스위치를 켠 값을 만든다.
- `~` (NOT): 모든 비트를 뒤집는다. `~0101 = 1010`. 켜져 있던 자리만 끄고 나머지를 켠 마스크를 만든다.
- `&` (AND): 양쪽 다 1인 자리만 남긴다. 여기서 `&`는 주소 연산자가 아니라 두 값 사이의 비트 AND다.
- `&=` (AND 후 대입): `A &= B`는 `A = A & B`의 줄임(`+=`와 같은 방식).

### 이 코드가 하는 일
```
  1111   (기존 tracemask, 모든 충돌 켜짐 가정)
& 1010   (~(CONTENTS_SOLID | CONTENTS_BOTCLIP), 그 두 자리만 0)
------
  1010   (그 두 자리만 꺼지고 나머지는 유지)
```
즉 tracemask에서 `CONTENTS_SOLID`와 `CONTENTS_BOTCLIP` 두 스위치만 끄고 나머지 충돌 설정은 그대로 둔다. `tracemask = 0`처럼 다 꺼버리면 유지해야 할 다른 설정까지 날아가므로, 원하는 비트만 정확히 끄는 `&= ~(...)` 관용구를 쓴다.

## 4. 향후 추가 예정
- 현재 주입기들은 봇·플레이어 모두에 적용된다. 실제 시스템에서는 `SVF_BOT` 검사로 봇에만 적용되도록 좁힌다.
- 바닥 관통 주입은 봇이 무한히 떨어지므로, 일정 깊이 이상 낙하 시 멈추거나 리스폰시키는 다듬기를 추가한다.
- 여러 주입기를 동시에 켜면 위치 주입끼리(oob, stuck 등) 충돌한다. 동시 사용이 필요해지면 우선순위/배타 처리를 정리한다.
- 과속(impossible_velocity)은 낙하 상황에서 검증됐다. 낙하가 아닌 상황에서 독립 검증하려면 `qa_inject_speed`를 추가한다(선택).
