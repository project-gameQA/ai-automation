# frametrace

게임 엔진용 프레임 프로파일러. 계측 매크로, 락프리 이벤트 수집, 바이너리 트레이스 포맷, 분석 도구로 구성된다.

C++17, 외부 의존성 없음. Linux/Windows/macOS.

```cpp
void update() {
    FT_SCOPE("update");
    update_ai();
    physics_step();
    render();
}
```

## 왜 만들었나

프레임이 평균 16ms인데 가끔 60ms로 튄다. 평균 FPS는 멀쩡해 보인다. 어느 함수가 범인인지 알려면 프레임 단위로 함수 호출 트리의 시간을 재야 한다. 이게 프로파일러가 하는 일이다.

문제는 재는 행위 자체가 비용이라는 것이다. 스코프 하나 재는 데 1us가 든다면, 프레임당 스코프 1000개는 1ms를 먹는다. 프레임 예산의 6%다. 측정이 측정 대상을 바꾸면 프로파일러가 아니라 거짓말 생성기가 된다. 그래서 이 프로젝트의 모든 설계 결정은 오버헤드를 낮추는 방향으로 정렬돼 있다.

## 측정된 오버헤드

Intel Xeon 2.10GHz, GCC 13.3, `-O2`, 단일 코어 컨테이너.

| 항목 | 비용 |
|---|---|
| `FT_SCOPE` 1개 (이벤트 2개) | 32.7 ns |
| 이벤트 1개 | 16.3 ns |
| 세션 비활성 시 | 1.95 ns |
| `FT_ENABLED=0` 컴파일 | 0 ns (측정 노이즈 이하) |

프레임 예산 환산 (60 FPS = 16.67ms):

| 프레임당 스코프 | 비용 | 예산 대비 |
|---|---|---|
| 1,000 | 0.033 ms | 0.20% |
| 10,000 | 0.329 ms | 1.97% |
| 50,000 | 1.644 ms | 9.86% |

이벤트당 16.3ns는 2.1GHz에서 약 34사이클이다. 이 중 대부분이 `rdtsc` 자체다. 즉 타임스탬프를 읽는 것 외에 하는 일이 거의 없다는 뜻이고, 이게 목표였다.

직접 재려면 `./bench_overhead`를 실행한다. 위 수치는 이 머신 기준이므로 자기 환경에서 다시 재야 한다.

## 사용법

```cpp
#include "frametrace/frametrace.h"

int main() {
    ft::Profiler::begin_session("out.ftrace");
    FT_THREAD_NAME("main");

    while (running) {
        FT_SCOPE("frame");
        update();
        render();
        FT_FRAME_MARK();
    }

    ft::Profiler::end_session();
}
```

매크로는 네 개다.

| 매크로 | 용도 |
|---|---|
| `FT_SCOPE("name")` | 스코프 시작~끝 자동 측정 |
| `FT_FUNC()` | 함수 이름으로 자동 측정 |
| `FT_FRAME_MARK()` | 프레임 경계 표시 |
| `FT_INSTANT("name")` | 지속 시간 없는 순간 이벤트 |
| `FT_THREAD_NAME("name")` | 스레드 이름 지정 |

`FT_ENABLED=0`으로 빌드하면 전부 사라진다. 런타임 분기가 아니라 전처리기 단계에서 제거되므로 출시 빌드에 남는 게 없다.

## 설계 결정

### 스레드당 락프리 SPSC 링

측정 대상 스레드는 절대 블로킹되면 안 된다. 락을 잡는 순간 재려던 타이밍이 왜곡된다.

스레드마다 링을 하나씩 준다. 그래서 MPMC가 아니라 SPSC로 충분하다. 생산자는 그 스레드 자신, 소비자는 플러시 스레드 하나뿐이다. SPSC는 CAS 없이 acquire/release store만으로 구현된다.

`head`와 `tail`은 각각 다른 캐시 라인에 놓는다. 같은 라인에 있으면 생산자와 소비자가 서로의 캐시 라인을 계속 무효화한다(false sharing). 여기에 더해 생산자는 `tail`을 캐싱해두고, 캐시가 오래됐을 때만 실제 `tail`을 읽는다. 정상 상황에서는 소비자의 캐시 라인을 아예 건드리지 않는다.

레이아웃이 성능에 직결되는데 성능 버그는 테스트로 안 잡힌다. 그래서 `sizeof`와 `alignof`를 테스트로 검증한다.

### 링이 가득 차면 이벤트를 버린다

블로킹하지 않는다. 프로파일러가 게임을 멈춰세우는 것보다 이벤트 몇 개를 잃는 게 낫다. 대신 버린 개수를 세서 트레이스 헤더에 기록하고, 뷰어가 경고를 띄운다. 조용히 잘못된 데이터를 보여주는 것이 최악이다.

### `rdtsc`, 캘리브레이션, fence 없음

`std::chrono::steady_clock::now()`는 리눅스에서 20~25ns다. 스코프당 두 번이면 40~50ns고, 이건 재려는 함수보다 비쌀 수 있다. `rdtsc`는 20~30사이클, 약 10ns다.

대신 단위가 사이클이다. 시작 시 10ms 동안 `steady_clock`과 대조해서 ticks/ns 비율을 구하고 트레이스 헤더에 넣는다. 뷰어가 이걸로 실제 시간을 복원한다.

`rdtsc`는 아웃오브오더 실행으로 재배치될 수 있다. `lfence`를 넣으면 막을 수 있지만 파이프라인이 비워져서 느려진다. 스코프가 수백 ns 이상이면 몇 사이클 흔들리는 건 무시할 수준이다. Tracy도 같은 선택을 한다. 정확도가 필요하면 `FT_SERIALIZE_TSC`로 켤 수 있다.

`rdtsc`가 없는 플랫폼은 `steady_clock`으로 자동 폴백한다.

### 문자열 인터닝을 포인터로

이벤트마다 이름 문자열을 넣으면 트레이스가 수백 MB가 된다. 32비트 ID로 참조하면 4바이트다.

`FT_SCOPE`에 넘기는 건 문자열 리터럴이므로 포인터가 고유하다. 내용 비교 대신 포인터로 조회한다. 매크로가 `static` 지역 변수로 ID를 캐싱하므로 인터닝은 스코프당 딱 한 번 일어난다. 이후 호출은 가드 변수 원자적 로드 한 번(약 1ns)으로 끝난다.

### 스레드가 죽어도 링을 해제하지 않는다

`thread_local` 객체는 스레드 종료 시 소멸된다. 그런데 플러시 스레드가 여전히 그 링을 가리키고 있을 수 있다. 소멸된 메모리를 읽으면 UB다.

링을 힙에 할당하고 전역 리스트가 소유한다. `thread_local`은 포인터만 들고 있고, 스레드가 죽으면 플래그만 내린다. 실제 해제는 `end_session()`에서 플러시 스레드를 조인한 뒤에 한다. 스레드당 256KB를 더 쓰지만, 프로파일러가 크래시를 내는 것보다 낫다.

### 이벤트 16바이트 고정

```cpp
struct Event {
    uint64_t timestamp;   // 원시 틱
    uint32_t name_id;
    uint16_t thread_id;
    uint8_t  type;
    uint8_t  _pad;
};
static_assert(sizeof(Event) == 16);
```

캐시 라인 하나에 4개가 들어간다. 패딩으로 크기가 늘면 링 처리량이 그만큼 떨어지므로 `static_assert`로 강제한다.

### 이벤트 먼저, 문자열 테이블 나중에

실행 중에는 어떤 문자열이 쓰일지 미리 알 수 없다. 헤더 자리를 비워두고 이벤트를 스트리밍으로 뱉다가, 종료 시점에 문자열 테이블을 덧붙이고 헤더를 되돌아가서 다시 쓴다.

바이너리인 이유는 단순하다. 초당 수백만 이벤트가 나온다. JSON 직렬화는 그 자체로 프로파일링 대상보다 비싸다.

## 벤치마크 방법론

오버헤드 측정에서 문제가 하나 있었고, 이걸 다루는 방식이 이 프로젝트에서 제일 중요한 부분일 수 있다.

처음에는 200만 개 스코프를 쉬지 않고 돌려서 쟀다. 스코프당 51ns가 나왔다. 그런데 드롭률을 찍어보니 **63%였다.** 링 용량은 16384개인데 초당 4000만 이벤트를 밀어넣고 있었으니 당연한 결과다.

드롭이 63%면 그 숫자는 대부분 "실패 경로 비용"이다. 알고 싶은 숫자가 아니다. 여기엔 반대 방향의 힘 두 개가 섞여 있다.

- 낮추는 쪽: 드롭 경로는 버퍼 쓰기와 release store를 건너뛴다. 성공 경로보다 싸다.
- 올리는 쪽: 플러시 스레드가 쉬지 않고 돌면서 CPU와 캐시를 놓고 경합한다.

이 머신(1코어)에서는 경합이 이겨서 포화 측정이 버스트보다 18ns 높게 나왔다. 코어가 많은 머신에서는 반대로 나올 수 있다. 어느 쪽이든 그 수치는 신뢰할 수 없다.

고친 방법은 링 용량보다 작은 버스트(4000 스코프)로 나눠 재고, 버스트 사이에 플러시 스레드가 비울 시간을 주는 것이다. 시간은 버스트 구간만 잰다. 드롭 0%가 되고 모든 push가 성공 경로를 탄다. 실제 게임도 스코프 사이에 진짜 일이 있으므로 이벤트가 초당 4000만 개씩 나오지 않는다. 버스트 측정이 오히려 현실에 가깝다.

벤치마크는 지금도 두 수치를 다 출력하고 드롭률과 해석을 함께 찍는다. 숫자 하나만 보여주는 벤치마크는 믿을 수 없다.

측정 전략:
- 평균이 아니라 최소값을 쓴다. 평균은 스케줄러 노이즈에 오염된다. 최소값이 "방해받지 않았을 때의 진짜 비용"에 가장 가깝다.
- `volatile` 싱크로 루프가 최적화로 사라지는 것을 막는다.
- 베이스라인(빈 루프)을 따로 재서 빼낸다.
- `FT_ENABLED=0` 바이너리를 별도로 빌드해서 "오버헤드 0"을 주장이 아니라 측정으로 보여준다.

## 분석 도구

`ft_dump`는 트레이스를 텍스트로 뽑는다. GUI와 별개로 CLI가 있는 이유는 CI에서 돌릴 수 있고, 포맷이 이상할 때 제일 먼저 볼 곳이기 때문이다.

```
$ ft_dump demo.ftrace --frames

== 프레임 시간 ==
  프레임 수 : 299
  평균      : 23.59 us
  p50       : 14.72 us
  p99       : 160.71 us
  최악      : 461.29 us (프레임 #38)

  [주목] p99가 p50의 10.9배다. 평균은 멀쩡한데 히칭이 있다.
```

평균만 보면 절대 안 보이는 문제를 p50/p99 비교로 잡아낸다. 이게 프로파일러의 존재 이유다.

```
$ ft_dump demo.ftrace --stats

이름                       호출       총합       평균       최소       최대
----------------------------------------------------------------------------
frame                       300     7.173 ms   23.91 us    8.80 us  461.22 us
physics_step                300     2.239 ms    7.46 us    2.76 us  102.90 us
asset_stream_hitch           15     1.637 ms  109.12 us   63.14 us  443.18 us
narrow_phase               2619     1.540 ms      588 ns      201 ns  91.22 us
```

`--json`으로 Chrome Trace Format으로 내보내면 `chrome://tracing`에 드래그해서 플레임 그래프로 볼 수 있다.

## 구조

```
include/frametrace/
    frametrace.h           공개 API, 매크로
    trace_format.h         바이너리 포맷 정의
    trace_reader.h         트레이스 로더, 스팬 복원
    detail/
        ring_buffer.h      락프리 SPSC 링
        clock.h            rdtsc, 캘리브레이션
src/
    frametrace.cpp         스레드 레지스트리, 인터닝, 플러시 스레드
    trace_reader.cpp       스팬 복원, 집계
tools/ft_dump/             CLI 분석 도구
demo/                      가짜 게임 루프
tests/                     링 버퍼 테스트, 오버헤드 벤치마크
```

이벤트 스트림은 Begin/End가 번갈아 나오는 평평한 목록이다. 플레임 그래프를 그리려면 스팬(시작, 끝, 중첩 깊이)으로 바꿔야 한다. 스레드마다 스택을 하나씩 두고 Begin에 push, End에 pop한다. pop할 때 스택 깊이가 그대로 중첩 깊이다.

## 알려진 한계

- **GPU 타임스탬프가 없다.** CPU만 잰다. 실무에서는 GPU 쪽이 병목인 경우가 많다.
- **메모리 할당 추적이 없다.** 할당 스파이크는 히칭의 흔한 원인이다.
- **실시간 뷰어가 없다.** 지금은 종료 후 파일을 읽는 방식이다. Tracy처럼 실행 중에 붙어서 보려면 소켓 전송이 필요하다.
- **`Instant` 이벤트의 depth가 부정확할 수 있다.** 현재 스택 깊이를 그대로 쓰는데, 드롭이 있으면 어긋난다.
- **스택 배열을 스코프 이름으로 쓰면 중복 항목이 생긴다.** 인터닝이 포인터 기반이라 그렇다. 문자열 리터럴만 쓰면 문제없지만 API가 이걸 강제하지 않는다.
- **트레이스 크기 제한이 없다.** 오래 돌리면 파일이 무한정 커진다. 링 버퍼 방식(최근 N초만 유지)이 필요하다.

## 다시 만든다면

- **인터닝을 컴파일 타임으로.** 지금은 `static` 지역 변수 가드 검사가 매 호출마다 원자적 로드를 한다. 약 1ns다. 소스 위치를 컴파일 타임 상수로 만들면 이것도 없앨 수 있다.
- **벤치마크를 처음부터 드롭률과 함께 설계했어야 한다.** 51ns라는 잘못된 숫자를 한동안 믿고 있었다. 지표를 하나만 보면 안 된다는 걸 실감했다.
- **`Event`에 `depth`를 넣는 걸 고려.** 지금은 읽을 때 스택으로 복원하는데, 드롭이 있으면 복원이 어긋난다. 4비트만 있어도 검증이 가능하다.
- **링 용량을 런타임 설정으로.** 지금은 컴파일 타임 상수다. 스레드마다 적정 크기가 다르다.
- **`FT_SCOPE` 중첩 깊이 제한.** 재귀 함수에 붙이면 스택이 무한정 커진다.
