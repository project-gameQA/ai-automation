GENERALIZED, NON-LLM VISION-BASED BUG DETECTION - STRATEGY SUMMARY
=======================================================================

CONTEXT
-------
Two constraints given:
1. The service should imitate an end-to-end product and NOT be tied to one specific game
   (game-agnostic).
2. Avoiding LLMs, and specifically avoiding supervised classifiers, because of the "too many
   bug categories + no labeled data" problem.

These two constraints actually solve each other, as laid out below.


WHY REQUIREMENT #1 CHANGES THE APPROACH
------------------------------------------
Engine-side ground truth (reading UE collision/bounds/NavMesh data directly) is NOT a good
default for a multi-client service, because it requires bespoke integration per client game -
it defeats the "generalize" goal. That approach only makes sense for testing your own single
game.

Pure screen-space vision generalizes better for a service, because pixels look the same
regardless of the underlying game logic - a floating object looks like a floating object
whether it's a boulder or a spaceship. This is why real products (e.g. Razer QA Companion-AI)
lead with "zero-integration" vision detection as the headline feature.

The one thing that DOES generalize cheaply at the engine level, with near-zero per-client
setup, is low-level engine telemetry that's identical across every UE game: FPS, memory,
crash signals, hang detection. This is the "free" layer every client gets automatically.


SOLVING THE DATA PROBLEM (REQUIREMENT #2) - TWO MOVES
----------------------------------------------------------

A. SYNTHETIC BUG INJECTION (the key move, specific to targeting Unreal only)
   Because the project targets UE only, the Unreal Editor Scripting API can be used to
   programmatically BREAK a scene and know exactly what was broken - giving pre-labeled data
   with zero manual annotation. Example injections via an Editor Utility script:
     - Swap a material to null -> triggers UE's default checkerboard/missing-material look
     - Disable a StaticMeshComponent's collision -> clips through geometry
     - Freeze an Animation Blueprint mid-blend -> T-pose / stuck pose
     - Set an actor's Z below the floor -> falls through world
     - Mismatch a mesh's collision bounds vs. visual bounds -> floating/misaligned object

   Since the injection script works against ANY UE project, this dataset can be generated
   across multiple free/sample UE projects (Marketplace samples etc.), producing a classifier
   that generalizes to games it's never seen - because defect signatures (checkerboard
   texture, floating silhouette, frozen pose) look structurally similar across different art
   styles. This is essentially "synthetic anomaly injection," a known technique in general
   defect-detection research, and it maps naturally onto Unreal's own editor tooling.

B. COLLAPSE BUG CATEGORIES INSTEAD OF ENUMERATING THEM
   Don't aim for dozens of specific per-game bug classes. Nearly all visual bugs reduce to a
   handful of generic, transferable signature classes:
     - Missing/placeholder texture
     - Clipping / z-fighting
     - Floating / out-of-bounds object
     - Animation freeze
     - UI overlap
   Train one classifier per signature class, not per game or per specific bug type.


C. TWO-STAGE PIPELINE - TIES GENERALIZATION AND DATA SOURCING TOGETHER
   -----------------------------------------------------------------------
   Stage 1 - UNSUPERVISED, per-client:
     An autoencoder calibrated on a short clip of NORMAL gameplay footage from that specific
     client's game (trivial to obtain - just a normal playtest recording, no bug labels
     needed). It flags "this frame looks statistically unusual for this game."
     -> Solves the per-client data problem: zero bug-labeled data required per client.

   Stage 2 - SUPERVISED, shared across ALL clients:
     Frames flagged by Stage 1 are passed to the synthetic-data-trained classifiers from (A)
     to determine WHAT KIND of anomaly it is (texture / clipping / animation / etc.).
     This model is trained once, on synthetic data, and reused for every client - it never
     needs client-specific data.

   Division of labor:
     - Generalization comes from Stage 2 being trained on synthetic, engine-agnostic-looking
       defects.
     - Data sourcing is solved because Stage 1 only needs normal (unlabeled) footage, and
       Stage 2's training data is generated programmatically, not manually collected.
     - Neither stage requires hunting for real labeled bug screenshots from real games.
